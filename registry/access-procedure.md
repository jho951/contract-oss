# 레포 접근 확인 절차

이 문서는 각 OSS / platform / service 레포에 대해 접근 가능 여부를 확인하는 절차를 정의한다.

## 목적

- 읽기 가능 여부를 확인한다.
- 클론 가능 여부를 확인한다.
- 쓰기 가능 여부를 확인한다.
- 각 레포에 실제 문서를 반영할 수 있는지 판단한다.

## 확인 순서

### 1. 존재 여부 확인

- GitHub에서 레포가 실제로 존재하는지 본다.
- 레포 이름, 대소문자, URL을 확인한다.

### 2. 읽기 권한 확인

- `git ls-remote <repo-url>` 로 원격 조회 가능 여부를 본다.
- `git clone <repo-url>` 로 로컬 복제가 가능한지 본다.

### 3. 쓰기 권한 확인

- 로컬 변경 후 `git push --dry-run` 으로 푸시 가능 여부를 확인한다.
- 실제 푸시는 사용자 승인 후 수행한다.

### 4. 브랜치 정책 확인

- 기본 브랜치 이름을 확인한다.
- 보호 브랜치 여부를 확인한다.
- PR 기반인지 직접 push 기반인지 확인한다.

### 5. 문서 반영 방식 확인

- 레포에 직접 문서를 추가할 수 있는지 본다.
- 직접 수정이 안 되면 패치 초안을 먼저 만든다.
- `CONTRACT_SYNC.md`를 기준 문서로 사용한다.

## 판정 기준

| 상태 | 의미 |
| --- | --- |
| 읽기 가능 | 문서 확인 가능 |
| 클론 가능 | 로컬 초안 비교 가능 |
| 쓰기 가능 | 실제 sync 가능 |
| 쓰기 불가 | 패치 초안만 준비 |

## 권장 운영

- 쓰기 가능하면 바로 sync한다.
- 쓰기 불가면 `sync-drafts/`와 `sync-templates/`를 기준으로 패치를 준비한다.
- 어떤 경우든 `oss-contract`를 기준 문서로 둔다.
