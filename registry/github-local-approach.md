# GitHub / Local 동기화 방식

이 문서는 레포를 실제로 동기화할 때 어떤 방식으로 GitHub와 로컬을 사용할지 정리한다.

## 원칙

- 문서는 로컬에서 먼저 준비한다.
- GitHub 반영은 마지막 단계다.
- 접근 권한이 없으면 패치만 준비한다.
- 접근 권한이 있으면 로컬 검증 후 반영한다.

## 권장 흐름

1. `oss-contract`에서 SOT 문서를 갱신한다.
2. `sync-templates/`에서 대상 레포용 `CONTRACT_SYNC.md`를 확인한다.
3. `sync-drafts/`에서 실제 레포에 넣을 README/docs 초안을 준비한다.
4. 대상 레포를 clone 한다.
5. 로컬에서 문구를 대조한다.
6. 직접 push가 가능하면 반영한다.
7. 직접 push가 불가능하면 PR/patch 형태로 전달한다.

## GitHub 사용

- 원격 존재와 브랜치 상태를 확인한다.
- 변경이 가능한 경우 PR을 선호한다.
- 문서 저장소라도 보호 브랜치를 존중한다.

## 로컬 사용

- 동기화 전 초안 검증
- 문구 diff 확인
- 경계 침범 확인
- 필요한 경우 패치 생성

## 패치 운영

- 권한이 없으면 `sync-drafts/`에만 두지 말고 실제 반영용 패치로 정리한다.
- 레포별로 파일 단위 패치를 만들면 나중에 적용이 쉽다.
